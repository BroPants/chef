 # Chef 微信小程序测试指南

> 版本：v1.0  
> 适用范围：当前已完成的 MVP（微信小程序前端 + Node.js/Express 后端）

---

## 1. 测试环境准备

- **后端服务**
  - 目录：`server/`
  - 环境：Node.js 18+，已安装依赖
  - 启动命令：

    ```bash
    cd server
    npm install
    npm run dev
    ```

  - 默认监听：`http://localhost:3000`

- **微信小程序**
  - 目录：`miniprogram/`
  - 使用微信开发者工具导入 `miniprogram` 目录
  - 开发者工具中勾选：**不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书**
  - 确保小程序中请求的后端地址指向本地后端（例如 `http://localhost:3000`，或通过开发工具代理）

- **配置**
  - 在 `server/.env` 中配置（本地测试可使用假值）：
    - `WECHAT_APPID`
    - `WECHAT_SECRET`

## 2. 功能测试用例（前端 + 后端联调）

### 2.1 微信登录（R-011）

- **用例 F-LOGIN-001：首次打开授权登录**
  - 前置条件：首次在该手机上打开小程序
  - 步骤：
    1. 打开小程序首页
    2. 触发登录入口（如首页自动调用或按钮触发）
  - 预期：
    - 小程序调用 `wx.login` 成功，后端 `POST /api/auth/login` 被请求
    - 后端正常返回 openid（或 mock 用户信息）
    - 小程序进入已登录状态（本地缓存登录态或用户信息）

- **用例 F-LOGIN-002：登录失败处理**
  - 模拟方式：关闭后端服务或让接口返回错误
  - 步骤：
    1. 打开小程序 / 点击登录
  - 预期：
    - 小程序展示明确的错误提示（例如「登录失败，请稍后重试」）
    - 不进入已登录状态

---

### 2.2 图片上传与菜品识别（R-001, R-002, R-003, R-004）

后端相关接口：
- `POST /api/upload`
- `POST /api/recognize`（当前为 mock：任意图片固定返回番茄炒蛋）

> 注意：当前 MVP 阶段识别结果为 mock，主要验证上传链路与前端展示是否正确。

- **用例 F-UPLOAD-001：拍照上传**
  - 步骤：
    1. 打开小程序首页
    2. 点击「拍照识别」按钮
    3. 授权相机权限，并实际拍摄一张菜品照片
  - 预期：
    - 能成功调起系统相机并完成拍照
    - 小程序展示上传中 / 加载中的状态
    - 后端 `POST /api/upload` 被调用，返回上传成功
    - 随后调用 `POST /api/recognize`，返回 mock 菜品（如番茄炒蛋）
    - 页面跳转到菜谱详情页，展示菜名、食材、步骤

- **用例 F-UPLOAD-002：相册选择上传**
  - 步骤与 F-UPLOAD-001 类似，将第 2 步改为「从相册选择」
  - 预期同 F-UPLOAD-001

- **用例 F-UPLOAD-003：识别失败提示**
  - 模拟方式一：临时修改后端 `recognize` 接口返回错误码 / 异常
  - 模拟方式二：关闭后端服务
  - 预期：
    - 前端出现友好的失败文案（例如「识别失败，请重试或更换图片」）
    - 页面可重新发起上传（如保留按钮）

### 2.3 菜谱展示页（R-006, R-007）

当前菜谱数据位于后端 `server/src/services/recipe.js`，内置若干道菜（含番茄炒蛋）。

- **用例 F-RECIPE-001：展示菜谱内容**
  - 步骤：
    1. 通过上传任意图片完成识别流程
    2. 进入菜谱展示页
  - 预期：
    - 页头显示菜品名称（与识别结果一致）
    - 食材列表以「食材 + 用量」形式展示
    - 制作步骤按顺序展示（每一步有完整说明）

- **用例 F-RECIPE-002：分享能力**
  - 步骤：
    1. 在菜谱展示页，点击右上角分享（或页面内分享按钮）
  - 预期：
    - 触发 `onShareAppMessage`，出现微信分享面板
    - 分享卡片标题或描述与菜谱相关

### 2.4 错误与空态

- **用例 F-STATE-001：网络异常**
  - 模拟方式：关闭网络 / 关闭后端
  - 步骤：
    1. 尝试登录或上传图片
  - 预期：
    - 显示网络错误提示（例如「网络异常，请检查网络后重试」）
    - 有明显的重新操作入口（重新上传 / 重新登录）

- **用例 F-STATE-002：空数据**
  - 模拟方式：让后端返回空的菜谱数据
  - 预期：
    - 页面展示空态文案（例如「暂无菜谱数据」）
    - 页面不崩溃，无白屏

## 3. 后端接口单独验证（API 层）

在前端联调之外，可直接使用 curl 或 Postman 访问接口，验证后端行为。

### 3.1 /api/upload

```bash
curl -X POST http://localhost:3000/api/upload \
  -F "image=@/path/to/your/image.jpg"
```

预期：
- 返回 HTTP 200
- 响应体中包含上传结果信息（如文件名、路径）

### 3.2 /api/recognize

```bash
curl -X POST http://localhost:3000/api/recognize \
  -H "Content-Type: application/json" \
  -d '{"imageUrl": "http://example.com/image.jpg"}'
```

预期：
- 返回 HTTP 200
- 响应体中包含菜品名称（当前为 mock：固定菜谱，如番茄炒蛋）及基础菜谱信息

### 3.3 /api/auth/login

```bash
curl -X POST http://localhost:3000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"code": "test-wechat-code"}'
```

预期：
- 返回 HTTP 200 或 4xx/5xx（视当前实现而定）
- 成功场景应返回 openid 或用户标识
- 失败场景应包含错误提示信息

## 4. 非功能性检查（MVP 级）

- **性能**
  - 本地连续上传 5–10 张图片，确认接口响应时间在可接受范围内（例如 < 2s，取决于本机环境）
  - 小程序页面切换流畅，无明显卡顿

- **稳定性**
  - 多次重复登录 / 上传 / 查看菜谱，应用不崩溃

- **日志与排查**
  - 后端在控制台输出关键日志（请求路径、错误信息）
  - 出现异常时，可根据日志快速定位问题

## 5. 后续自动化测试规划（占位）

当前 MVP 阶段暂未引入自动化测试框架，建议后续按以下方向扩展：

- 在 `server/` 中引入 Jest / Vitest，用于编写接口层单元测试与集成测试
- 在小程序端引入页面级自动化回归（例如使用小程序端自动化测试工具）
- 为关键功能（登录、上传、识别、菜谱展示）补充回归用例

本文件作为当前测试的基础说明，可在功能扩展时继续补充详细用例与自动化脚本链接。
